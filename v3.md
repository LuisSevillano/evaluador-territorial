Fase 3 — Analítica de decisión y experiencia de uso
Objetivo general

Pasar de un visor funcional con datos fiables a una herramienta de lectura territorial útil para el grupo, reforzando tres cosas:

interpretación de los datos
filtros realmente útiles
transparencia metodológica

Todavía no entraría en scoring final completo si no queréis, pero sí dejaría todo preparado para ello.

Prioridad 1: mejorar el valor de la ficha municipal

Ahora que ya tenéis clima real e isocronas, la ficha municipal debe convertirse en una pieza central.

Qué añadir
resumen climático claro:
precipitación anual
precipitación mensual
temperatura media anual
media invierno / verano
mínima típica invierno
máxima típica verano
accesibilidad:
bucket de isocrona
etiqueta legible: “≤ 2h”, “2h–2h30”, etc.
señales rápidas:
“clima seco / medio / húmedo”
“verano suave / caluroso”
“accesibilidad alta / media / baja”
Qué evitar
sobrecargarla con demasiados números crudos
gráficos bonitos pero poco interpretables
Resultado esperado

Una ficha que permita leer un municipio en 20–30 segundos.

Prioridad 2: filtros duros realmente potentes

Ahora mismo ya podéis pasar de filtros básicos a filtros útiles de verdad.

Añadir filtros por:
precipitación anual mínima
temperatura media verano máxima
temperatura media invierno mínima
bucket máximo de isocrona
provincia
rango altitudinal, si lo incorporáis pronto
UX recomendada
sliders o selectores simples
botón “limpiar filtros”
contador visible de municipios restantes
texto tipo:
“Mostrando 84 de 2.248 municipios”
Resultado esperado

Que el grupo pueda pasar de miles de municipios a una lista razonable sin esfuerzo.

Prioridad 3: tabla/listado analítico de municipios

Aunque no queráis aún comparador formal, sí os conviene un listado ordenable.

Qué incluir
nombre
provincia
isocrona
precipitación anual
temp. invierno
temp. verano
Funciones mínimas
ordenar por columna
abrir ficha al pulsar
exportar selección o resultados filtrados
Por qué es importante

Porque el mapa solo no basta para decidir.
Necesitáis una vista de “lectura comparativa ligera”.

Prioridad 4: trazabilidad visible en la interfaz

Ahora que el pipeline ya es serio, conviene que la app lo refleje.

Añadir en UI
fuente del clima
periodo climatológico usado
método de agregación municipal
definición de isocronas
fecha de generación del dataset
versión de datos
Formato

Un bloque discreto de “Metodología” o “Ficha técnica”.

Resultado esperado

Más confianza interna y menos discusiones difusas sobre “de dónde sale esto”.

Prioridad 5: preparar la capa de normalización

Aunque no lancéis aún scoring final, yo dejaría lista la base.

En pipeline

Generar columnas como:

precip_norm
temp_verano_norm
temp_invierno_norm
accesibilidad_norm

No hace falta usarlas todavía en la UI principal.

Por qué ahora

Porque:

os obliga a cerrar bien el sentido de cada variable
deja todo preparado para scoring futuro
ayuda a detectar anomalías
Prioridad 6: shortlist o favoritos

Esto no es todavía “comparador”, pero sí una función muy útil.

Funcionalidad

Permitir:

marcar municipios
guardar una shortlist
revisar luego solo los seleccionados
Valor real

Esto conecta mucho mejor con el uso grupal que un mapa puro.
